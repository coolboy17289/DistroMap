# DistroMap

![DistroMap logo](assets/logo.svg)

A visual knowledge graph of the Linux ecosystem.

DistroMap maps **383 Linux distributions** across **47 families** and
shows how they connect — starting from the **Linux kernel** and branching
out through Debian, Arch, Fedora, Gentoo, Slackware, openSUSE, Alpine,
Void, NixOS, and dozens more. Click a distro to see its full lineage,
package manager, init system, release model, and every child it has
spawned.

The whole stack is one TypeScript codebase. One `npm install`, one
`npm run dev`, one process. The same code deploys to Vercel as a single
project (root = `frontend/`) with the React SPA + the API in the same
Vercel origin. The build step (Python) lives inside `frontend/scripts/`
and runs automatically on `npm run dev` / `npm run build` via
`predev` / `prebuild` hooks — so a fresh clone of `frontend/` plus an
`npm install` is enough to run the whole app.

## What ships

- **Radial graph** of 383 distros (255 active, 127 discontinued) — every
  record has a slug, family, parents, status, package manager, init
  system, architectures, license, country, and 12 boolean category flags
  (server / gaming / immutable / rolling / LTS / security / privacy /
  education / embedded / container / cloud / arm).
- **Side panel** with the full record on click: status badge, all
  parents, direct children, version history, package format, license,
  maintainer, Wikipedia link, OG image, share to X.
- **Family legend bar** under the header — every family as a clickable
  chip, color-coded, with the distro count. Click to filter the canvas
  to that family; double-click to focus on the family root. Press `f`
  anywhere on the page to jump to the legend. Press `Esc` to clear.
- **Auto-search** with smart filters:
  - Bare tokens (`server`, `gaming`, `rolling`, `debian`, `canonical`).
  - `key:value` filters: `family:debian`, `status:discontinued`,
    `parent:arch`, `package:apt`, `init:systemd`, `arch:aarch64`,
    `country:Germany`, `dev:canonical`, `license:GPL`.
  - All tokens are AND-ed; multiple filters narrow the result.
- **Random button** picks a random distro (`/api/random`).
- **Theme toggle** (light / dark) with a strict 3-color palette per
  theme. Distro brand color tints the node border at low alpha so
  the structural palette stays dominant.
- **URL state**: `?d=arch` opens the arch side panel, `?f=debian`
  filters to debian family. Refresh / share-link both work.
- **Suggestion form** so users can submit a missing distro. The form
  stores the suggestion in Vercel KV in production or in a local file
  in dev; a localStorage + downloadable-JSON fallback covers the case
  where the API is offline.

## API

All read-only endpoints return JSON. No API keys are required. The
underlying data is rebuilt at build time only, so responses are
edge-cacheable.

| Method & path | What it returns |
|---|---|
| `GET  /api/health` | Health check + dataset summary |
| `GET  /api/distros` | Full list of distros (filterable via `?family=`, `?status=`, `?category=`) |
| `GET  /api/distros/:slug` | One distro by slug |
| `GET  /api/search?q=foo&limit=10` | Ranked search across name, slug, description, family, country, package manager, desktop |
| `GET  /api/families` | Every family with count, active count, and root slug |
| `GET  /api/family/:family` | Every distro in a family, ordered by depth then name |
| `GET  /api/related/:slug` | Distros in the same family at any depth |
| `GET  /api/descendants/:slug` | Full descendant tree (slugs + slugs-of-slugs) |
| `GET  /api/ancestors/:slug` | Ancestor chain back to linux_kernel |
| `GET  /api/path?from=foo&to=bar` | Shortest parent-chain path between two distros |
| `GET  /api/compare?a=foo&b=bar` | Two distros side by side + their common ancestor |
| `GET  /api/stats` | Counts: total, active, discontinued, per family, per category, per release model, per package format, per init system |
| `GET  /api/random` | One random distro |
| `GET  /api/graph` | The `graph.json` (nodes + edges) directly |
| `GET  /api/categories` | List of category flags + counts per flag |
| `GET  /api/og/:slug.svg` | Dynamic OG image (SVG) — distro name + family + 1-line desc |
| `GET  /api/og/:slug.png` | Same as above, but with `Content-Type: image/png` for clients that only accept PNG |
| `POST /api/suggestions` | Submit a missing distro. Stored in Vercel KV in prod, local file in dev. |
| `GET  /api/suggestions` | Read existing suggestions (admin / debug) |

All GET responses carry `Cache-Control: public, s-maxage=3600,
stale-while-revalidate=86400` so Vercel can edge-cache them.

### Examples

```bash
curl https://distromap.app/api/health
curl https://distromap.app/api/distros/ubuntu
curl 'https://distromap.app/api/search?q=server&limit=5'
curl 'https://distromap.app/api/path?from=arch&to=endeavouros'
curl https://distromap.app/api/random
curl https://distromap.app/api/og/ubuntu.svg   # shareable OG image
```

## How the data is built

Every distro lives in a slice file under
`frontend/src/data/_slices/<family>.json`. One record per distro,
flat shape, all the metadata. The synthesis step
(`frontend/scripts/synthesize.py`) does, in order:

1. **Load** every slice.
2. **Dedupe** by slug (keep the most-detailed record).
3. **Validate** parent references — drop any parent that doesn't exist,
   attach `linux_kernel` to any record with no remaining parents.
4. **BFS from `linux_kernel`** to compute `depth` for every record.
5. **Compute `children[]`** for every record (the inverse of `parents`).
6. **Build `graph.json`** — one node per distro, one edge per parent→child.
7. **Compute polar angles** for the radial layout, write `layout.json`.
8. **Emit `distros.json`** with the full record shape (parent, parents,
   children, depth, derived legacy fields like `favicon_url`,
   `wiki_url`, `short_desc`).

To regenerate after editing a slice:

```bash
cd frontend
npm run synthesize      # or: python3 scripts/synthesize.py
```

Output is written to `frontend/src/data/{distros,graph,layout}.json`.
The dev server picks the JSON up on next request (no rebuild needed).
`npm run dev` and `npm run build` both invoke the synthesis step
automatically via `predev` / `prebuild` hooks, so a fresh checkout only
needs `npm install` before it can serve the latest dataset.

## Deploying

DistroMap ships on **Vercel free tier**, full-stack, in **one** project
that has **`frontend/` as the Vercel project root directory**.

| Layer | Where it runs | Free-tier constraints |
|-------|---------------|----------------------|
| Static Vite SPA    | **Vercel** (`frontend/dist`) | 100 GB/mo bandwidth |
| API (18 endpoints) | **Vercel Node function** (`frontend/api/index.ts`, `@vercel/node`) | 100 GB-hr/mo, 10s execution |
| Suggestion queue   | **Vercel KV** (Upstash Redis REST) | 256 MB, 30K commands/mo |

The SPA and the API share the same Vercel origin. The
`frontend/vercel.json` rewrite `/api/(.*) → /api/index` routes every
`/api/*` request to the single TypeScript serverless function, so no
CORS config is required for same-origin calls.

### One-time setup

1. Pull the repo on GitHub, import it in Vercel's "Add New Project"
   flow. Set the **Project Root Directory** to **`frontend/`** in the
   "Build & Development Settings" step.
2. From the project dashboard, open **Storage → Create Database → KV**.
   Vercel auto-injects `KV_REST_API_URL` and `KV_REST_API_TOKEN` into
   the Production + Preview environments; the API picks them up on
   cold start and switches from file mode to KV mode automatically.
3. **File mode will 500 on production POSTs.** Vercel's serverless
   filesystem is read-only (only `/tmp` is writable); without KV
   linked, the very first POST to `/api/suggestions` will fail with a
   filesystem error. Link KV.

### Deploys

```bash
# from the repo root, with the Vercel project already configured
vercel --prod                 # production deploy

# every PR / branch gets its own preview URL automatically
vercel                        # preview deploy
```

### Custom domain

`distromap.app` (or any custom domain) just works — set it under
**Settings → Domains**. The `ALLOWED_ORIGINS` env var
(comma-separated) extends the CORS allowlist beyond the default
`*.vercel.app` regex; add your custom domain here on the backend.

### Locally simulating deploy

```bash
cd frontend
npm run dev      # Vite on :5173 — serves the SPA AND /api/* in-process
                 # (no separate backend, no second port, no Python)
```

Without KV env vars the API uses the local
`frontend/.cache/api/suggestions.json` path (file mode) with a
promise-chain mutex around read-modify-write so concurrent POSTs
don't lose rows. Once you set `KV_REST_API_URL` +
`KV_REST_API_TOKEN` (e.g. via a `.env` file) and restart `npm run dev`,
the same API switches to KV mode without code changes.

### Why one TypeScript function at `frontend/api/index.ts`?

The `vercel.json` rewrite `/api/(.*) → /api/index` covers every API
path with a single Vercel function, no dead endpoints. The same file
runs unchanged in local dev — the `apiServerPlugin` in
`vite.config.js` loads it via Vite's SSR module graph
(`server.ssrLoadModule`) and calls its default export directly
against the dev server's req/res. One command (`npm run dev`),
one process, one port, one language.
